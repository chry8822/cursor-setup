# Commit

아래 순서대로 커밋을 진행한다.

## 진행 순서

### STEP 1. 변경 파일 파악
터미널에서 아래 명령어를 실행해 현재 변경 사항을 파악한다.
```
git status
git diff HEAD
```

### STEP 2. 커밋 타입 선택 안내
STEP 1 결과를 바탕으로 변경 내용을 한 줄로 요약한 뒤,
아래 형식으로 타입 선택지를 출력한다.

```
변경된 파일 목록:
- src/components/main/Profile.tsx
- src/constants/localStorage.ts
- ...

──────────────────────────────────
커밋 타입을 선택하세요:

  1. feat     — 새로운 기능 추가
  2. fix      — 버그 수정
  3. refactor — 기능 변경 없이 코드 구조 개선
  4. style    — 코드 포맷, 세미콜론 등 스타일 변경 (로직 변경 없음)
  5. chore    — 설정, 패키지, 빌드 관련 변경
  6. docs     — 주석, README 등 문서 변경
  7. test     — 테스트 코드 추가/수정
  8. 직접 입력 — 위 항목에 없는 타입 직접 입력

번호 또는 타입명을 입력하세요:
──────────────────────────────────
```

### STEP 3. 커밋 메시지 생성
사용자가 타입을 선택하면, STEP 1의 diff 내용을 분석해
아래 형식으로 커밋 메시지 초안을 출력한다.

```
──────────────────────────────────
생성된 커밋 메시지:

  feat: 커뮤니티 프로필 컴포넌트 리팩토링 및 localStorage 상수화

변경 요약:
- Profile.tsx: handleGoToDetail 함수 추출, Fragment 제거, useMutation 타입 명시
- localStorage.ts: COMMUNITY_LOCAL_STORAGE_KEYS 상수 추가

이 메시지로 커밋할까요?
  y        → 이 메시지로 커밋 진행
  n        → 커밋 취소
  수정할 내용 직접 입력 → 입력한 내용으로 메시지 교체 후 커밋
──────────────────────────────────
```

### STEP 4. 커밋 실행
사용자가 확인하면 아래 명령어를 순서대로 실행한다.
```
git add .
git commit -m "타입: 메시지"
```

커밋 완료 후 아래 형식으로 push 여부를 묻는다.

```
✅ 커밋 완료

[브랜치명 커밋해시] 타입: 메시지
N files changed, ...

──────────────────────────────────
push도 진행할까요?
  y  → git push 실행
  n  → push 없이 종료
──────────────────────────────────
```

### STEP 5. Push (선택)
사용자가 y를 입력하면 아래를 실행한다.
```
git push
```

실행 결과를 출력하고 종료한다.

## 규칙
- push는 반드시 사용자 확인 후에만 실행한다
- 커밋 메시지는 한국어 또는 영어 모두 허용 (기존 커밋 메시지 스타일을 따른다)
- 메시지 생성 시 diff 전체를 기반으로 작성하며, 추측으로 작성하지 않는다
- 변경 파일이 없으면 커밋하지 않고 "변경된 파일이 없습니다"를 출력하고 종료한다

## 속도 팁
diff 분석과 메시지 생성만 하는 커맨드이므로 실행 전 Cursor 모델을 빠른 모델로
전환하면 응답 속도가 크게 개선된다. (claude-3.5-haiku, gpt-4o-mini 등)
